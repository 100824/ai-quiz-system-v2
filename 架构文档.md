# 架构文档

本文档面向后续维护者，尤其是 Claude Code。目标不是重复 README，而是把“项目为什么长这样、关键链路在哪、改一个功能该进哪几个文件、哪些地方最容易踩坑”讲清楚。

## 1. 项目定位

这是一个课堂内使用的人工智能学习平台，核心使用场景是：

- 教师选择当前课程与班级
- 教师控制课堂进行到哪个环节
- 学生按班级和姓名进入课堂答题
- 教师实时查看课堂完成情况和统计结果
- 学生课后回看自己的历史课堂答题数据

系统在交互上有明显的“课堂时序”特征，因此它既不是单纯的问卷系统，也不是通用 LMS，而是带有课堂实时进度控制的教学活动系统。

## 2. 当前总体架构

当前正式链路：

```text
浏览器
  -> frontend/student.html / teacher.html / student-history.html
  -> frontend/js/*.js
  -> HTTP 调用 backend-go/cmd/server/main.go 提供的 /api/*
  -> SQLite: backend-go/data/quiz-system.db
```

历史链路：

```text
backend/server.js
backend/database.js
```

这个历史 Node 后端仍保留在仓库中，但已经不是默认维护对象。

Go 后端已按以下分层组织：

```text
backend-go/cmd/server/main.go          -> 启动入口、路由注册
backend-go/internal/models/            -> 数据模型
backend-go/internal/utils/             -> 通用工具函数
backend-go/internal/middleware/        -> CORS、日志中间件
backend-go/internal/migrations/        -> 数据库初始化与迁移
backend-go/internal/repository/        -> 数据库 CRUD
backend-go/internal/service/           -> 业务逻辑（统计、导出、Part2 处理）
backend-go/internal/handler/           -> HTTP handlers
```

## 3. 为什么会同时存在 Go 后端和 Node 后端

项目经历过“旧平台迁移 + Go 重构”：

- 旧 Node 后端保留在 `backend/`
- 新 Go 后端实现放在 `backend-go/`
- 现在前端默认对接的是 Go 后端

保留历史 Node 后端的主要价值：

- 对照旧行为
- 查询旧实现思路
- 必要时做兼容性参考

但日常维护应遵循一个原则：

> 除非明确是修历史遗留部署脚本，否则默认改 `backend-go/`，不要把新逻辑继续加到 `backend/`。

## 5. 当前前后端协作方式

### 5.1 前端如何找到后端

前端不会写死 API 地址，而是按当前页面的协议和主机名拼出：

```text
${window.location.protocol}//${window.location.hostname}:8080/api
```

实现位置：

- [frontend/js/config.js](/Users/fuhaotong/Documents/class_system/frontend/js/config.js:1)

这意味着：

- 前端运行在本机 `127.0.0.1:3000` 时，请求 `127.0.0.1:8080`
- 前端运行在局域网 IP `10.x.x.x:3000` 时，请求相同 IP 的 `8080`

这个设计是为了项目迁移到其他机器时，不需要再次硬编码 IP。

### 5.2 前端页面状态

学生端和教师端都不是 SPA 框架项目，而是“一个 HTML + 一个主 JS”的结构。

这意味着：

- 页面状态多靠全局变量维护
- 功能间耦合主要出现在 `student.js` / `teacher.js`
- 修改某个功能时，要同时关注“渲染 + 提交 + 回显 + 已提交锁定 + 统计兼容”

## 6. 业务模型

### 6.1 课程

一节课对应一条 `courses` 记录。

字段上最重要的是：

- `id`
- `name`
- `lesson_number`
- `part2_guide`

### 6.2 班级

班级不是课程的附庸，而是被“班级名 -> 当前课程”绑定。

相关表：

- `class_lists`
- `class_course_bind`

当前约定：

- 学生名单按班级共享
- 不因为切到第二节课就重新维护一份名单

这是之前修过的关键业务规则，后续不要轻易改回“按课程维护名单”的模式。

### 6.3 课堂阶段

课堂不是单纯按“答了第几部分”推进，而是教师显式控制。

相关表：

- `course_stages`

当前阶段包括：

- `0`：准备环节
- `1`：第一部分
- `2`：第二部分
- `3`：第三部分
- `4`：第四部分

相关接口入口：

- [backend-go/cmd/server/main.go](/Users/fuhaotong/Documents/class_system/backend-go/cmd/server/main.go:649)

### 6.4 题目

相关表：

- `questions`

当前题目支持字段：

- `question_type`
- `options`
- `correct_answer`
- `explanation`
- `enabled`
- `annotation_enabled`
- `sort_order`

其中两个很重要的新增字段：

- `enabled`
  控制题目是否启用
- `annotation_enabled`
  控制第二部分填空题是否要求颜色标注

### 6.5 学生答题

相关表：

- `student_surveys`

当前存储方式：

- `part1_answers`：JSON
- `part2_answer`：JSON / 富文本结果
- `part3_answers`：JSON
- `part3_score`：整数
- `part4_answers`：JSON

这里最大的维护点在 `part2_answer`：

- 它经历过结构升级
- 需要兼容旧数据和新数据
- 同时服务于学生端回显、教师端统计、历史页展示、导出

- 支持单选

- 支持富文本填空
- 支持可选的颜色标注要求
- 支持至少一次标注校验
- 支持历史回显
- 支持教师端统计
- 支持 Excel 导出

- 多行标注按“原地切分文本节点”处理
- 不再通过整段 `extractContents()/insertNode()` 的方式重组换行

## 9. 数据库表职责

当前最重要的几张表：

- `courses`
  课程元数据与第二部分答题指引
- `class_lists`
  班级学生名单
- `class_course_bind`
  班级绑定课程
- `course_stages`
  课堂阶段
- `course_part_settings`
  每节课各部分是否启用
- `questions`
  各部分题目配置
- `student_surveys`
  学生作答结果
- `system_config`
  当前课程等系统级配置

一个容易忽略的点：

`questions` 表是“按课程分题目”的，因此迁移历史数据时必须留意：

- 是否正确保留了各课程自己的题目
- 是否错误复用了第一节课的题目模板

这类问题之前出现过一次，是交接时要特别警惕的。

## 10. 当前技术债

### 10.1 ~~Go 后端单文件过大~~（已拆分）

Go 后端已从单文件 `main.go` 拆分为 `models / utils / middleware / migrations / repository / service / handler` 分层。后续维护时，按职责定位文件即可。

### 10.2 ~~PM2 配置仍指向旧 Node 后端~~（已修正）

[ecosystem.config.js](/Users/fuhaotong/Documents/class_system/ecosystem.config.js:1) 已更新为启动 Go 编译后的二进制 `backend-go/server`。

使用前需要先执行 `npm run backend:build` 生成可执行文件。

### 10.3 数据库迁移依赖启动时自动执行

当前 Go 后端在启动阶段直接 `ALTER TABLE`。

优点：

- 新环境上手快

缺点：

- 数据库状态不透明
- 二进制数据库文件经常被动变化
- 交接时不容易判断哪些改动是结构、哪些改动是数据

建议后续引入显式 migration version 管理。

### 10.4 前端是全局变量式状态管理

这套写法能跑，但需要维护者养成一个习惯：

> 每改一个功能，都要检查“初始化、提交、回显、锁定、历史展示、统计导出”是不是同时兼容。

## 12. 运行与维护建议

### 12.1 本地调试

推荐双进程：

1. 前端静态服务跑 `3000`
2. Go 后端跑 `8080`

这是当前最贴近真实使用的模式。

### 12.2 问题定位顺序

遇到 bug 时建议按这个顺序排查：

1. 先看前端页面是否渲染异常
2. 再看对应接口是否返回正确 JSON
3. 再看数据库里是否已有预期字段或记录
4. 最后检查是不是旧 Node 链路、旧 PM2 配置或旧数据结构干扰

### 12.3 提交策略

如果改动涉及：

- 前端逻辑
- Go 接口
- 数据库文件

最好分开提交，便于后续回溯。

## 14. 一句话总结

这个项目的真正难点不是“页面或接口单独怎么写”，而是：

> 它是一个带课堂阶段控制、富文本答题、历史数据兼容、统计导出联动的教学系统。维护时必须始终把“前端交互、接口协议、数据库结构、历史数据兼容”四件事绑在一起看。
